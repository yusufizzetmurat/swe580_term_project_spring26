# Prompt Optimizer

Automated prompt optimization for the [[Chatbot_Prototype]] and [[RAG_System]]. Manual prompt engineering is slow and brittle — this project explores systematic prompt search and compilation.

## Motivation

After [[Industry_Talk_Jan29]], I started thinking more seriously about prompt robustness. The speaker mentioned that production systems at Google iterate through thousands of prompt variants in automated pipelines. We're doing it by hand, which doesn't scale.

## Approach: DSPy-style Compilation

Inspired by the DSPy framework. Instead of hand-writing prompts, define the task as a signature (input fields → output fields) and let an optimizer find the best prompt automatically.

The optimizer runs a few-shot example selector and a prompt proposer, evaluates candidate prompts against a labeled development set, and keeps the best performing variant.

## Search Space

For each prompt component:
- Instruction phrasing (5-10 candidates generated by the LLM itself)
- Number of few-shot examples (0, 1, 3, 5)
- Example selection strategy (random, BM25 retrieval, embedding similarity)

Full grid search is expensive — using a greedy search that optimizes one component at a time.

## Evaluation

Tested on [[Chatbot_Prototype]] question-answering. Optimized prompts improved answer accuracy from 72% to 81% on our annotated test set. Most of the gain came from better few-shot example selection — domain-specific examples matter more than instruction phrasing.

Results logged in [[Experiment_Tracker]] under `promptopt_v1` through `promptopt_v4`.

## Limitations

- Optimized prompts are specific to the model they were optimized on. Switching to a different model requires re-running the optimizer.
- The development set needs to be representative — optimizing on a biased set produces prompts that overfit to that distribution.
- Compute cost: each optimizer run calls the LLM ~200 times
